# 운명PT 프로젝트 작업 규칙

이 문서는 운명PT 프로젝트에서 Claude Code, Cursor, 기타 AI 코딩 도구가 반드시 따라야 하는 고정 작업 규칙이다.

운명PT는 단순 사주 사이트가 아니라,
**AI 자기이해 리포트 기반의 무료 결과 → 잠금 해금 → 결제 업셀링 퍼널**이다.

모든 코드 수정은 아래 원칙을 기준으로 진행한다.

---

# 1. 프로젝트 정체성

## 1-1. 서비스명
* 서비스명: 운명PT
* 외부 포지셔닝: AI 자기이해 리포트
* 내부 상품 설명: 정통 사주 기반 AI 자기이해 리포트

## 1-2. 핵심 태그라인
* "왜 나는 같은 선택을 반복할까?"

## 1-3. 서비스 성격
운명PT는 다음 구조를 가진다.
1. 사용자가 무료 진단을 시작한다.
2. 이름, 생년월일, 성별, 태어난 시간, MBTI 등을 입력한다.
3. 무료 결과 일부를 확인한다.
4. 잠긴 상세 리포트와 블러 처리된 프리미엄 분석을 본다.
5. 더 자세한 결과를 보고 싶어 결제한다.
6. 결제 후 전체 리포트가 해금된다.

핵심은 사용자가 "상품을 구매한다"가 아니라
**"내 결과의 다음 장을 해금한다"**고 느끼게 만드는 것이다.

# 2. 이름 표시 정책

## 2-1. 기본 원칙
고객 결과 페이지에서는 절대 특정 실명, 특히 "장진아"를 고객명처럼 하드코딩하지 않는다.
고객명은 반드시 입력값 기반으로 동적 치환한다.

```ts
const displayName = userName ? `${userName}님` : "고객님";
```

## 2-2. 고객 결과 페이지 이름 표시 규칙
아래 영역에서는 반드시 `displayName`을 사용한다.

* `/results/[id]`
* PersonaHero
* FREE_BODY
* LockedItems
* PremiumPreview
* Paywall
* StickyUnlockCTA
* UnlockProgress
* 모든 고객 결과 본문
* 모든 고객용 CTA
* 모든 잠금 카드
* 모든 무료 결과 텍스트

## 2-3. 올바른 표현
* `${displayName}의 사주 원본 분석`
* `${displayName}께 새겨진 여덟 글자의 운명`
* `${displayName}의 재물·관계·일·실천 전략`
* `${displayName} 전체 리포트 열기`
* `${displayName}의 결과 이어보기`

## 2-4. 금지 표현
* "장진아님의 사주 원본 분석"
* "장진아님께 새겨진…"
* "장진아님 전체 리포트 열기"
* 고객 결과 페이지 내 "장진아" 하드코딩 전체 금지

## 2-5. "장진아" 표시 허용 영역
다음 영역에서는 "장진아"가 법적/결제 정보로 유지되어야 한다.

| 영역                          | 처리 | 이유    |
| --------------------------- | -- | ----- |
| `src/config/site.ts` 사업자 정보 | 유지 | 법적 표시 |
| `/checkout` 무통장입금 예금주       | 유지 | 결제 정보 |
| `/legal/*` 대표자              | 유지 | 법적 고지 |
| `/legal/*` 개인정보책임자          | 유지 | 법적 고지 |

주의:
사업자 정보의 "장진아"와 고객 결과 페이지의 고객명은 절대 혼동하지 않는다.

# 3. 퍼널 구조

운명PT의 핵심 전환 구조는 아래 순서를 따른다.

```text
무료 진단
→ 무료 결과 일부 공개
→ 명식/성향/점수 확인
→ "내 얘기 같다"는 몰입 형성
→ 잠금 카드 노출
→ 추가 무료 해석
→ 다시 잠금 카드 노출
→ 프리미엄 블러 미리보기
→ 해금 진행률 표시
→ 결제 업셀링
→ 결제 후 전체 리포트 해금
```

이 순서를 임의로 생략하거나 축약하지 않는다.

# 4. 무료 결과 페이지 텍스트 흐름

무료 결과 페이지는 반드시 아래 흐름을 가진다.

## 4-1. Hero
역할:
사용자가 "나만을 위한 결과가 열렸다"고 느끼게 한다.

필수 요소:
* `${displayName}의 사주 원본 분석`
* 유형명
* 유형 부제
* CTA: "내 리포트 보기"

예시:
```text
고객님께 도착한 사주 원본 분석
자유영혼형
누구의 틀에도 갇히지 않는 독보적 자유인
```

## 4-2. 명식표 / 원본 구조
역할:
결과가 그냥 심리 테스트가 아니라, 입력값 기반으로 분석되었다는 신뢰를 준다.

포함 요소:
* 년주
* 월주
* 일주
* 시주
* 오행
* 십성
* 신강/신약
* 격국
* 대운/세운 가능 시 표시

시주가 없으면 "시주 미입력"으로 처리한다.

## 4-3. 점수/지표 카드
역할:
사용자가 "분석받고 있다"는 느낌을 받게 한다.

예시 지표:
* 예리함
* 집중력
* 판단력
* 독립성
* 직관
* 재물감각
* 사교 감각

주의:
* 과학적 진단처럼 단정하지 않는다.
* "성향 리포트 지표" 느낌으로 표현한다.

## 4-4. 무료 해석 1: 첫 몰입
제목 예시:
```text
${displayName}께 새겨진 여덟 글자의 운명
```

역할:
* 사용자가 "내 얘기 같다"고 느끼는 첫 몰입 구간
* 너무 짧으면 안 됨
* 전체 분석 중 일부만 열린 느낌을 줌

내용 방향:
* 겉으로는 차분하고 신중해 보임
* 안쪽에는 자기 기준이 분명함
* 남의 길보다 스스로 납득한 방식으로 움직일 때 힘이 살아남
* 재물·관계·일의 흐름은 전체 리포트에서 이어짐

## 4-5. 무료 해석 2: 타고난 기질
제목 예시:
```text
얼어붙은 대지 위에서 스스로 빛나는 보석
```

내용 방향:
* 신중함과 예리함
* 쉽게 흔들리지 않음
* 결정적 순간에는 자신의 판단을 믿고 움직임
* 감정 표현은 조심스럽지만 내면 기준은 강함
* 이 성향이 재물과 관계에서 어떻게 작동하는지는 전체 리포트에서 이어짐

## 4-6. 잠금 카드 1
무료 해석 중간에 반드시 잠금 카드를 넣는다.

문구 예시:
```text
${displayName}의 보석 같은 감각이 현실에서 빛나는 결정적 순간
전체 명식에서 확인 가능
```

역할:
* "뒤에 더 중요한 내용이 있다"는 결핍 생성
* 무료 결과와 유료 리포트 사이의 다리 역할

## 4-7. 무료 해석 3: 살아가는 방식
제목 예시:
```text
필터 없는 예리함이 만드는 현실적 통찰
```

내용 방향:
* 복잡한 상황 속에서도 핵심을 빠르게 짚어냄
* 사람과 흐름을 날카롭게 파악함
* 다만 모든 것을 혼자 판단하고 책임지려 할 때 피로가 쌓임
* 전체 리포트에서는 이 패턴이 돈, 관계, 일에서 어떻게 반복되는지 분석함

## 4-8. 잠금 카드 2
문구 예시:
```text
당신의 재능이 막히는 지점과 다시 흐름을 여는 방법
프리미엄 리포트에서 확인 가능
```

## 4-9. 프리미엄 블러 미리보기
카테고리 예시:
```text
부의 흐름
```

제목 예시:
```text
보석의 전략 — 재물이 폭발하는 순간
```

공개 문장 예시:
```text
${displayName}의 재물 구조는 평소에는 신중하게 내공을 쌓다가, 결정적인 타이밍에 자신의 예리한 안목을 투여해 성과를 거두는 구조입니다.
```

아래 본문은 블러 처리한다.

블러 처리 원칙:
* 실제 내용이 더 있는 것처럼 보여야 한다.
* `display: none` 금지
* `filter: blur(...)` 사용 가능
* gradient overlay 사용 가능
* 중앙에 자물쇠 아이콘 또는 해금 안내 표시
* 문구: "전체 명식 분석에서 확인할 수 있습니다"

## 4-10. 해금 진행률
필수 문구:
```text
리포트 해금
2 / 22 항목
프리미엄 구매 시 22개 항목 전체 해금
```

카테고리 예시:
* 재물의 기질 0/4
* 일과 커리어 0/5
* 관계의 기술 0/4
* 실전의 시작 0/3

역할:
* 이미 일부를 봤다는 느낌
* 아직 더 많이 남았다는 결핍
* 결제를 "전체 해금"으로 인식시키는 장치

## 4-11. 결제 업셀링
결제 유도 카드는 구매형이 아니라 해금형이어야 한다.

문구 예시:
```text
무료 결과는 시작일 뿐입니다.
${displayName}의 재물·관계·일·실천 전략 전체를 확인해보세요.
결제 후 잠긴 22개 항목이 즉시 해금됩니다.
```

가격 예시:
* 정가 54,900원
* 오늘 할인 34,900원
* 36% OFF · 오늘만
* 12,000+명이 먼저 받아본 리포트

CTA 예시:
* `${displayName} 전체 리포트 열기 →`
* `잠긴 22개 항목 전체 해금하기 →`
* `내 결과 이어보기 →`

# 5. CTA 문구 원칙

## 5-1. 금지 문구
고객 결과 페이지, Paywall, Sticky CTA에서 아래 문구를 사용하지 않는다.

* 결제하기
* 구매하기
* 상품 구매
* 정통 사주 구매하기
* 지금 구매
* 상품 결제

## 5-2. 권장 문구
* 전체 리포트 열기
* 잠긴 항목 해금하기
* 내 결과 이어보기
* 전체 결과 보기
* 프리미엄 리포트 열기
* 잠긴 22개 항목 전체 해금하기

## 5-3. 핵심 원칙
사용자가 "돈을 낸다"가 아니라
**"내 결과의 다음 장을 연다"**고 느끼게 한다.

# 6. 결제 후 해금 로직

## 6-1. 핵심 원칙
결제 완료 사용자는 결과 페이지에서 모든 잠금이 풀려야 한다.

## 6-2. 기준
결제 완료 여부는 `paid_at` 또는 주문 상태 기준으로 판단한다.

## 6-3. 무료 사용자 화면
무료 사용자에게는 다음이 보인다.

* 무료 결과 일부
* 잠금 카드
* 프리미엄 블러
* 해금 진행률
* Paywall
* Sticky CTA

## 6-4. 유료 사용자 화면
유료 사용자에게는 다음이 숨겨져야 한다.

* Paywall
* Sticky CTA
* PremiumPreview 블러
* UnlockProgress
* 결제 유도 CTA

유료 사용자에게는 다음이 보여야 한다.

* 전체 리포트 본문
* 잠금 해제된 상세 분석
* 결제 완료 상태에 맞는 안내

## 6-5. 치명 버그 기준
결제했는데도 잠금이 보이면 치명 버그다.
이 경우 디자인 작업보다 해금 로직 수정이 최우선이다.

# 7. 로그인/게스트 정책

무료 진단 퍼널은 전환율을 위해 게스트 진입을 허용한다. 단, 게스트 허용은 `/start`, `/api/free`, 무료 결과 조회, 결제 업셀링, resultId 기반 결제 해금까지로 제한한다. 마이페이지, 구매내역, 결과 보관함, 계정 기반 관리 기능은 로그인/계정 연결이 필요하며 별도 도메인 작업으로 분리한다.

## 7-1. 게스트 허용 범위 (퍼널 전환 우선 — 정식 정책, 예외 아님)

로그인 없이 다음 흐름이 동작해야 한다.

* `/start` — 5단계 입력 퍼널 진입
* `/api/free` — 무료 진단 생성
* `/results/[resultId]` — 무료 결과 페이지 조회
* `/products/[slug]` 진입 — 결제 업셀링 진입
* 결제 완료 후 resultId 기반 해금

이유:
운명PT의 핵심 퍼널은 **광고/콘텐츠 유입 → 무료 진단 즉시 체험 → 무료 결과 확인 → 잠금/블러 → 결제 업셀링** 구조다.
무료 진단 시작 전 로그인 강제는 전환율을 떨어뜨린다. 따라서 무료 진단은 frictionless entry로 운영한다.

## 7-2. 로그인 필수 영역 (계정 기반 관리)

다음 영역은 로그인/계정 연결이 필수다. 게스트는 진입 불가하거나 별도 안내.

* `/mypage` — 마이페이지
* `/mypage/orders` — 구매내역
* `/mypage/reviews` — 후기 작성/관리
* 결과 보관함 (계정에 묶인 과거 결과 리스트)
* `/admin/*` — 관리자
* 고객 DB 직접 접근 (서비스 측 키 필요)

## 7-3. 데이터 연결 기준

* 무료 결과는 `resultId` 기준으로 생성/조회한다.
* 결제는 `resultId` 또는 `orderId`와 연결한다.
* 결제 완료 시 `paid_at` 또는 주문 상태 기준으로 전체 리포트를 해금한다.
* **로그인 사용자가 아니어도 결제한 resultId는 정상 해금되어야 한다.**
* 게스트 주문은 `orders.user_id = null` 로 저장한다 (DB 컬럼 nullable).
* 로그인 사용자 주문은 `orders.user_id = auth.uid()` 로 연결한다.

## 7-4. 게스트 모드 운영 원칙

* 게스트 모드는 무료 진단 퍼널 전환을 위한 **정식 정책**이다 (즉흥 변경이 아님).
* 전체 인증 구조를 갈아엎는 작업이 **아니다** — 무료 진단 입구만 게스트 허용.
* 카카오 로그인, 회원가입 간소화, 마이페이지 결과 저장 연동은 **별도 실험/별도 도메인 작업**으로 분리한다.
* 어뷰징 신호 관찰 시 IP 기반 rate limit 등 별도 보호 장치는 추가 가능 (현재 미적용).

## 7-5. 구현 체크리스트

게스트 모드 정책을 따르는지 확인.

* [ ] `/start` 페이지가 비로그인 상태에서 5단계 입력 가능한가
* [ ] `/api/free` 가 비로그인 요청에 401 을 반환하지 **않는가**
* [ ] `StartFunnel.submit()` 이 비로그인 시 `/login` 으로 리다이렉트하지 **않는가**
* [ ] `orders.user_id` 가 nullable 인가 (게스트 주문 저장 가능)
* [ ] `/results/[resultId]` 가 비로그인 게스트 접근 시에도 결과를 표시하는가
* [ ] 결제 완료 후 게스트도 `paid_at` 기준 잠금이 해제되는가

# 10. 입력값 반영 원칙

무료 진단 입력값은 결과에 실제로 반영되어야 한다.

## 10-1. 이름
* 모든 고객용 문구에 `displayName` 반영
* 이름이 없으면 "고객님"

## 10-2. 생년월일/성별/태어난 시간
* 만세력 계산에 반영
* 년주/월주/일주/시주가 입력값에 따라 달라져야 함
* 시주 미입력 시 안전하게 처리

## 10-3. MBTI
MBTI는 단순 저장값이 아니라 해석 보정에 사용해야 한다.

반영 가능 영역:
* 성향 해석
* 관계 패턴
* 일하는 방식
* 선택 패턴
* MbtiSajuMirrorCard
* 무료 결과 본문 일부

## 10-4. 고정 템플릿 금지
이름만 바뀌고 본문이 모두 같은 결과는 실제 서비스 전 반드시 개선해야 한다.

# 11. 환경변수 / API 검증 원칙

`/api/free`가 실패하면 결과 페이지 검수보다 API 정상화가 먼저다.

## 11-1. 필수 환경변수
Vercel 환경변수에 아래 값이 있어야 한다.

```text
LLM_PROVIDER=openai
LLM_MODEL=gpt-4o-mini
```

## 11-2. ZodError 대응
`ZodError invalid_type expected string` 발생 시 확인할 것:

* `LLM_MODEL`
* `LLM_PROVIDER`
* `birthDate`
* `birthTime`
* `gender`
* `mbti`
* `name`

선택 입력값은 `null` 또는 `undefined`로 보내지 말고 안전하게 처리한다.

# 14. 검수 기준

프리뷰 검수 시 아래를 확인한다.

## 14-1. 이름 치환
* 테스트 이름이 결과 전체에 반영되는가?
* 고객 결과 페이지에 "장진아"가 없는가?
* 법적 표시/예금주 영역과 고객 결과 영역을 혼동하지 않았는가?

## 14-2. 텍스트 흐름
필수 흐름:

```text
유형 히어로
→ 명식표
→ 점수/지표
→ 무료 해석 1
→ 무료 해석 2
→ 잠금 카드 1
→ 무료 해석 3
→ 잠금 카드 2
→ 프리미엄 블러 카드
→ 해금 진행률
→ 후기
→ 페이월
→ 모바일 Sticky CTA
```

## 14-3. CTA
* "결제하기/구매하기" 금지 문구가 없는가?
* "열기/해금/이어보기" 톤인가?

## 14-4. 무료 사용자
* 잠금 카드가 보이는가?
* 블러 카드가 보이는가?
* Paywall이 보이는가?
* Sticky CTA가 보이는가?

## 14-5. 유료 사용자
* 전체 리포트가 해금되는가?
* Paywall이 숨겨지는가?
* Sticky CTA가 숨겨지는가?
* 블러가 사라지는가?

---

# 15. 절대 금지

아래 행동은 하지 않는다.

* 고객 결과 페이지에 "장진아" 하드코딩
* 사업자 정보와 고객명 혼동
* 전체 디자인 리뉴얼을 임의로 진행
* 기존 컴포넌트 전면 재구축
* 텍스트 흐름 축약
* 무료 결과를 너무 짧게 만들기
* 잠금 카드를 맨 아래에만 몰아넣기
* 결제 CTA를 "구매" 톤으로 작성
* 사용자 승인 전 main 머지
* 빌드 실패 상태에서 배포 진행
* 결제 후 잠금 해제 버그 방치

---

# 16. 한 줄 요약

운명PT의 핵심은
**무료 결과로 몰입시키고, 잠금/블러로 결핍을 만들고, 결제를 "구매"가 아니라 "내 결과 해금"으로 느끼게 하는 퍼널**이다.

모든 작업은 이 원칙을 기준으로 판단한다.
